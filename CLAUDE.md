# Project Rules — AI-QA-POC

These rules apply to every session and every task in this project.
Claude must follow them automatically without being asked each time.

---

## RULE 1 — Dedicated Folder per Requirement / Execution

**Every time a new requirement, bug retest, or test suite is executed, create a dedicated
folder at the project root.** Never scatter files across the root or generic directories.

### Folder naming
| Type | Name format | Example |
|---|---|---|
| Requirement-based | `{ShortName}-E2E` | `nopCommerce-Guest-Checkout-E2E` |
| Bug retest | `BUG-Retest-{FeatureName}` | `BUG-Retest-Insurance-Name-Truncation` |
| Feature test | `{FeatureName}-Tests` | `PaymentGateway-Tests` |

### Required sub-folder structure inside every dedicated folder

```
{FolderName}/
├── README.md                  ← Always create this — navigation guide + run command
├── requirement/               ← Source requirement doc + any supporting notes
├── test-cases/                ← Human-readable TC list (markdown table)
├── scripts/                   ← Playwright spec files (sub-folders by suite if needed)
├── page-objects/              ← Page Object Model files for this feature
├── fixtures/                  ← Playwright fixture files (if using fixture pattern)
├── test-data/                 ← Test data files (JSON, TS constants)
├── config/                    ← Playwright config for this suite
└── output/                    ← All execution results (never outside this folder)
    ├── run-history/           ← One entry per execution (see Rule 2)
    ├── playwright-report/     ← Playwright HTML reports
    └── artifacts/             ← Screenshots, videos, traces (auto-created)
```

Optional sub-folders (add only when content exists):
- `tools/` — utility scripts specific to this suite
- `architecture/` — design decisions and SOPs

## RULE 2 — New Timestamped Sub-folder on Every Re-run

**When re-running an existing test suite, NEVER overwrite previous results.**
Instead, create a new timestamped folder inside `output/run-history/`.

### Timestamp format: `YYYYMMDD-HHMM`

```
output/
├── run-history/
│   ├── 20260528-1151/         ← First run
│   │   └── report.html
│   ├── 20260528-1432/         ← Second run (same day, different time)
│   │   └── report.html
│   └── 20260529-0930/         ← Re-run next day
│       └── report.html
└── playwright-report/
    └── latest/                ← Always the most recent run (overwritten)
```

The `output/playwright-report/latest/` folder is always replaced with the newest run.
The `output/run-history/` folder accumulates entries — never delete them.

### HTML report naming
Each individual run generates a standalone HTML file named:
`{suite-name}-{YYYYMMDD}-{HHMM}.html`
Example: `insurance-retest-20260528-1151.html`

## RULE 3 — README.md is Mandatory

Every dedicated folder MUST have a `README.md` that includes:
1. One-line description of what the folder is for
2. Folder structure diagram with purpose of each sub-folder
3. Exact `npx playwright test` command to run this suite
4. Table: folder name → what to open it for

---

## RULE 4 — No Files at Project Root

The project root (`AI-QA-POC/`) should only contain:
- Framework infrastructure: `src/`, `playwright.config.ts`, `tsconfig.json`, `.env`, `package.json`
- Dedicated project folders: `nopCommerce-Guest-Checkout-E2E/`, `BUG-Retest-Insurance-Name-Truncation/`, etc.
- This `CLAUDE.md` file

**Do NOT place** requirement docs, spec files, reports, screenshots, or test data
directly at the project root or in generic folders like `output/`, `reports/`, `tests/`.

---

## RULE 5 — Shared Framework vs Project-Specific Files

### Stays in `src/` (shared, do not move)
- `src/base/BasePage.ts` — all page objects extend this
- `src/utils/` — logger, screenshot manager, test data manager
- `src/helpers/` — assertion, string, wait helpers
- `src/config/` — environment config
- `src/models/` — shared TypeScript interfaces
- `src/constants/` — app-wide constants
- `src/api/` — API health check utilities

### Goes into the dedicated folder (project-specific)
- Requirement / spec documents
- Test spec files
- Page objects specific to the feature under test
- Playwright fixtures for that feature
- Test data
- Playwright config
- All execution outputs (reports, screenshots, JSON results)

---

## RULE 6 — Output File Standards

| File | Where it goes | When to create |
|---|---|---|
| Per-run HTML report | `output/run-history/{YYYYMMDD-HHMM}/` | After every single execution |
| Playwright HTML report | `output/playwright-report/latest/` | After every execution (overwrite) |
| Iteration summary HTML | `output/run-history/iteration-results.html` | After 2+ runs (compare all) |
| JSON results | `output/test-results.json` | Auto-generated by config |
| Screenshots | `output/artifacts/` | Auto-generated by Playwright |

---

## RULE 7 — HTML Report Format (mandatory, every run)

**Every per-run HTML report (`{suite}-{YYYYMMDD}-{HHMM}.html`) MUST be a custom
standalone dashboard — never a raw copy of the Playwright `index.html`.**

### Required sections (in order)

| Section | What to include |
|---|---|
| Header | Suite name, run date, start time, RUN_ID, environment URL, filename |
| Result Banner | Green (all pass) or amber (any fail/warn) gradient with icon, summary line |
| Stats row | TCs Passed · TCs Failed · TCs Skipped · Hook status · Total duration |
| Visual Summary | SVG donut chart (pass/fail) + per-suite progress bars + execution timeline |
| Test Results table | TC ID · Test case description with suite chip · Priority · Badge · Timestamp range |
| Acceptance Criteria | Table: criterion → test coverage → MET/NOT MET badge |
| Environment card | URL, user, browser, Playwright version, config file, run ID |
| Footer | Timestamp, RUN_ID, links to iteration-results.html and previous run |

### Style rules (must match the established format)
- **Background:** `#f0f4f8` body, white cards, `border-radius: 12px`, subtle `box-shadow`
- **Pass colour:** `#38a169` green — badges, progress bars, timeline dots, donut arc
- **Fail colour:** `#e53e3e` red — badges, bar segments
- **Warn colour:** `#d69e2e` amber — hook warnings, infrastructure notes
- **Suite chips:** `TS-001` → blue, `TS-002` → green, `TS-003` → amber, `TS-004` → pink, `TS-005` → purple, `TS-006` → orange
- **Donut chart:** SVG inline, 42×42 viewBox, `r="15.9155"`, `stroke-dasharray` encodes pass%
- **Progress bars:** 10px height, green fill, one row per test suite
- **Fonts:** `-apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif`
- **Code snippets:** monospace, `background:#edf2f7`, `border-radius:4px`
- **File size:** target ~20–30 KB (custom HTML only — no embedded Playwright assets)

### What NOT to do
- **Never** copy `output/playwright-report/latest/index.html` as the run report — that is 500KB+ of bundled Playwright assets and is unreadable without its `/data/` folder
- **Never** create a report with a different layout than the established format
- The Playwright `index.html` is kept separately in `output/playwright-report/` for deep-dive debugging; it is NOT the run report

### Reference template
Use `BUG-Retest-Insurance-Name-Truncation/output/run-history/insurance-retest-20260528-1151.html`
as the canonical template for format and sections.
